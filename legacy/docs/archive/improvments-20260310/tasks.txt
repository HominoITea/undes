Для ЗАДАЧИ 1
  я GPT-5 Codex, предлагаю следующее: внедрить двухэтапный пайплайн "индексация -> точечная выборка контекста". На этапе индексации (офлайн/инкрементально) парсить код через AST (например, `ts-morph`/TypeScript Compiler API для TS/JS, аналоги для других языков), строить граф зависимостей `symbol -> symbol` (calls, imports, extends/implements, type refs), сохранять в `code_index/` (JSON/SQLite) с координатами `file:startLine:endLine`, сигнатурами и хешами. На этапе запроса не отправлять целые файлы: 1) определить целевые символы по задаче (по имени + полнотекстовый поиск + опционально embeddings), 2) расширить выборку на N-hop соседей в графе (обычно 1-2), 3) вырезать только нужные диапазоны строк (функция/метод + минимальный контекст), 4) собрать "Context Pack" (сигнатуры, выбранные фрагменты, список связей), и только его отправлять в LLM. Для актуальности обновлять индекс по `git diff`/mtime, а для динамических вызовов добавить опциональный runtime trace (инструментация в test/dev) и объединять его со статическим графом. Это даст резкое снижение токенов, при этом сохранит достаточный контекст для точных ответов.

Для ЗАДАЧИ 1 я Claude Opus 4.6, предлагаю следующее:

Согласен с общим направлением GPT-5 Codex (AST-индексация + граф + точечная выборка), но вижу критический архитектурный момент: этот проект позиционируется как **универсальный тулкит** для любого стека (Node.js, .NET, Python, Java), а AST-парсеры языко-специфичны. Полноценный мульти-язычный AST — это отдельный проект уровня Language Server Protocol. Предлагаю прагматичный подход в 3 фазы, встроенный в существующий `generate-context.js`:

**Фаза 1 (zero-dependency, встраивается в текущий скрипт):** Regex-based Symbol Index. При сборке бандла извлекать сигнатуры символов (функции, классы, методы, экспорты) регулярными выражениями для основных языков — набор паттернов: `function \w+`, `class \w+`, `def \w+`, `public .* \w+\(` и т.д. Результат — файл `.code_index.json` с записями `{symbol, file, line, endLine, type, signature}`. Это не полный AST, но покрывает 80% случаев без зависимостей. Граф вызовов на этом этапе строится через grep-подобный поиск: если символ `UserService` упоминается в файле `controller.ts` — это ребро графа. Индекс обновляется инкрементально по `mtime` файлов (механизм хешей уже есть в `computeFilesHash`). Новый раздел в `ai/context.json`:

```json
{
  "codeIndex": {
    "enabled": true,
    "languages": ["ts", "js", "py", "cs", "java"],
    "maxDepth": 2,
    "outputPath": ".code_index.json"
  }
}
```

**Фаза 2 (prompt-aware выборка):** Перед отправкой в LLM добавить этап `resolveContext(promptText, codeIndex)`. Алгоритм: 1) извлечь из промпта ключевые слова и имена символов (простой NLP: split + фильтр стоп-слов + match по индексу); 2) найти прямые совпадения в `.code_index.json`; 3) расширить на 1-2 hop по графу вызовов; 4) вырезать только диапазоны строк `file:startLine-endLine` (функция `readFile` в скрипте уже читает файлы — добавить параметр `{fromLine, toLine}`); 5) сформировать секцию `## RELEVANT CODE FRAGMENTS` в бандле вместо полных файлов. Это встраивается между шагами "файлы из fullFiles" и "directory structure" в `main()` (строки 2060-2066 скрипта).

**Фаза 3 (опциональное усиление):** Подключить `tree-sitter` (есть биндинги для Node.js, поддержка 100+ языков из одной библиотеки) для точного AST вместо regex. Это единственная зависимость, но она покрывает все целевые языки одним инструментом — в отличие от подхода "ts-morph для TS, ast для Python, Roslyn для C#". `tree-sitter` парсит файл за миллисекунды, выдаёт дерево с типами узлов и координатами. Заменяет regex-паттерны из Фазы 1 без изменения остальной архитектуры.

**Ключевые отличия от предложения Codex:**
1. **Фаза 1 работает без зависимостей** — regex-индекс можно внедрить за 1 день без npm install, что соответствует духу проекта ("copy into your project and go").
2. **Единый `tree-sitter` вместо N парсеров** — для Фазы 3 одна зависимость покрывает все языки, а не отдельный парсер на каждый стек.
3. **Embeddings избыточны на старте** — для проектов до ~500 файлов keyword matching по индексу + граф вызовов дают достаточную точность. Embeddings добавляют latency, зависимость от API и стоимость. Стоит рассматривать только для монорепозиториев >1000 файлов.
4. **SQLite избыточен** — JSON-файл с lazy-load достаточен для индекса до ~10K символов. Проект уже использует JSON для кэша (`.context_cache.json`), добавление SQLite ломает zero-dependency философию.
5. **Runtime trace нецелесообразен** — требует инструментации кода пользователя, что противоречит позиции "универсального тулкита". Статический граф + grep-поиск упоминаний покрывает 90% связей.

**Ожидаемый эффект:** Фаза 1+2 сократят code-контекст на 70-90% для типовых задач ("рефакторни функцию X", "добавь метод в класс Y"), при этом внедряются в существующий `generate-context.js` без новых зависимостей и без ломающих изменений API скрипта.

Для ЗАДАЧИ 1 я Gemini 3 Pro Preview, предлагаю следующее:

Поддерживаю вектор развития в сторону графовой индексации (Codex) и прагматичного внедрения через `tree-sitter` (Claude), но считаю, что для достижения максимальной эффективности не хватает слоя **Intent-Based Context Orchestration**. Индексация — это "как искать", но нам нужно также определить "что именно нужно для конкретного типа задачи". Предлагаю дополнить архитектуру следующим:

**1. Гибридный фильтр намерений (Intent Filter):** Первым шагом в `generate-context.js` внедрить микро-классификатор (на основе дешевой модели или набора эвристик), который разбивает задачу на типы: `REFAC` (рефакторинг), `FEAT` (новая фича), `BUG` (исправление). 
- Для `REFAC`: Фокус на 2-hop зависимости и сигнатуры типов.
- Для `FEAT`: Фокус на схемы данных (SQL/Prisma) и существующие паттерны из `ai/PATTERNS.md`.
Это позволит динамически менять "глубину" и "состав" `Context Pack`.

**2. Virtual File System (VFS) для контекста:** Вместо физического вырезания строк, формировать в памяти структуру "виртуального файла", где тело функций заменяется на комментарии `// [implementation hidden, see index]`, если они не относятся к прямому графу вызовов. Это сохраняет "чувство структуры" файла для LLM (она видит класс целиком), но потребляет на 80% меньше токенов.

**3. Двухуровневый кэш (Context Pinning):** 
- **L1 (Session Cache):** Использование `cache_control: { type: 'ephemeral' }` для фиксации уже переданных фрагментов при уточняющих вопросах.
- **L2 (Project Index):** Метаданные о "частоте правок" (из git). Файлы, которые меняются чаще всего, должны индексироваться глубже.

**4. Автоматическая дозагрузка (Self-Healing Context):** Если агент `Tester` обнаруживает ошибку компиляции из-за отсутствующего типа, оркестратор автоматически подтягивает этот модуль из индекса и запускает повторный раунд без участия пользователя.

**Ожидаемый эффект:** Снижение галлюцинаций за счет VFS и уменьшение объема данных еще на 40% поверх AST-индексации за счет семантической фильтрации по типу задачи.
