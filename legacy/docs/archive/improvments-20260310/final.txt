Рабочая редакция (согласовано всеми)

Цель
Сократить токены за счет передачи в LLM только релевантных фрагментов кода вместо полных файлов, без потери качества ответов.

1) Базовая архитектура
- Внедрить пайплайн `Index -> ResolveContext -> Context Pack`.
- `Index`: инкрементальный индекс символов и связей (`calls`, `imports`, `extends/implements`, `type refs`) с координатами `file:startLine:endLine`.
- `ResolveContext`: выбор целевых символов по запросу + расширение на 1-2 hop по графу.
- `Context Pack`: только нужные сигнатуры, связи и вырезки диапазонов строк.

2) Этапы внедрения

P0 (MVP, без новых зависимостей)
- Добавить regex-индексацию в `ai/scripts/generate-context.js`.
- Хранить индекс в `.code_index.json`.
- Обновлять индекс инкрементально (`mtime`/`git diff`).
- Формировать контекст из диапазонов строк, а не из полных файлов.
- Ввести лимиты `maxPackBytes` и `maxGraphEdges`.
- Fallback: если индекс недоступен, использовать текущий режим (`fullFiles`/`lightFiles`) без ошибки.

P1 (точность)
- Подключить `tree-sitter` как единый AST-движок.
- Сохранить тот же формат `.code_index.json` и `Context Pack` (без ломки API).
- Улучшить точность связей и покрытие мульти-язычных проектов.

P2 (условные интеллектуальные надстройки)
- Intent filter (`REFAC`/`FEAT`/`BUG`) только если замеры покажут пользу.
- Map-Reduce Search только при низкой уверенности индексатора.
- VFS-режим только для зависимостей 2+ hop.
- Self-healing только однократный (max 1 дополнительный раунд), предпочтительно на фазе Tester.

3) Обязательные guardrails
- Для 0-1 hop всегда передавать полную реализацию целевых методов/функций.
- При скрытии реализации (VFS) обязательно оставлять:
  - type signature,
  - JSDoc/docstring (если есть),
  - файл и диапазон строк.
- Любой "умный" шаг (intent/map-reduce/self-healing) должен иметь hard-limit по числу дополнительных вызовов.

4) Метрики и критерии перехода между фазами
- Набор из 5-10 типовых задач как benchmark.
- Измерять:
  - Input tokens/run,
  - Precision/Recall релевантных фрагментов,
  - latency run,
  - число доп. вызовов.
- Переход P0 -> P1 и P1 -> P2 только при подтвержденной выгоде по метрикам.

5) KPI
- Снижение входных токенов: 30-50% минимум на типовых задачах.
- Целевой диапазон после стабилизации: 70-90% на задачах с локальной областью изменений.
- Без деградации качества решений по benchmark-набору.

6) Что делать сейчас
- Шаг 1: реализовать P0 полностью.
- Шаг 2: снять метрики "до/после" на benchmark-наборе.
- Шаг 3: при достаточной выгоде перейти к P1.
