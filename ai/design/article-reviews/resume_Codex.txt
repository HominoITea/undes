## Short Verdict

Главный вывод: нам не нужно просто "больше агентов". По orchestration и runtime discipline проект уже сильнее большей части описанных подходов. Основные пробелы сейчас в другом:

1. Долговременная семантическая память.
2. Spec-first слой между запросом и реализацией.
3. Надёжный structured edit protocol вместо обычных текстовых правок.
4. Взвешивание качества critique/review, а не только agreement.
5. Canonical documentation compiler как отдельный продуктовый слой.

## Что у нас уже реализовано лучше

- Multi-agent debate, critique, consensus, disagreement handling. Большинство статей описывают single-agent или UX-уровень; у нас уже есть production-like multi-stage pipeline.
- Runtime hardening. Уже есть `prompt-gate`, `response-validator`, `=== END OF DOCUMENT ===`, checkpoint resume, result cache, delta context, adaptive context budget, quality feedback loop.
- Project-scoped hub и strict dispatcher. У нас сильнее дисциплина входа, project resolution и audit trail.
- Typed logs и reproducibility. Мы уже лучше в явных артефактах, чем решения, завязанные на "помнит чат" или на UI-состояние.
- `ai:init` уже закрывает bootstrap части проблемы: проектный скан, контекстный scaffold, базовые конфиги.
- Context pack / code index уже сильнее, чем у многих ad-hoc agent workflows, особенно с tree-sitter fallback и hybrid indexing.

## Где статьи показывают более сильное решение, чем у нас сейчас

### 1. Память

Это наш самый заметный gap.

Сейчас у нас:
- `ai:memory` и typed logs
- snapshots по логам
- архивы прогонов

Но у нас нет:
- semantic retrieval по прошлым решениям
- typed memory model: facts / decisions / episodes / open questions
- cross-session durable memory, пригодной для реального recall
- общего memory API/MCP слоя для разных клиентов

Статьи `996154`, `994618` и `1001368` здесь заметно впереди.

### 2. Spec-first workflow

Сейчас у нас хорошая execution pipeline, но ещё нет обязательного spec layer перед кодом.

Есть база:
- prompt quality gate
- `ai:init`
- запланированные `jira-take` и `docs:compile`

Но пока нет жёсткого потока:

`intent -> spec -> plan -> tasks -> implementation -> validation`

Здесь сильнее статьи `995466` и частично `995038`.

### 3. Structured edit protocol

Сейчас мы просим safe/minimal diffs и валидируем ответы, но у нас нет нормального edit contract уровня hashline/anchor references.

Это значит:
- правки всё ещё чувствительны к формату ответа модели
- нет дешёвого pre-apply reject по несовпадению anchors
- нет fallback matrix по формату правок под конкретную модель

Статья `995986` здесь даёт очень сильную и прикладную идею.

### 4. Critique quality weighting

Сейчас у нас есть critique phase и agreement score. Это уже хорошо.

Но у нас пока нет:
- ранжирования полезности замечаний
- разделения signal vs noise
- weighting critiques по specificity / code refs / severity
- использования качества critique как входа в consensus

Статья `1000718` явно даёт полезный direction.

### 5. Operator UX

Если мы когда-то пойдём в TUI/UI слой, то мы слабее решений в духе `1000976`:
- нет explicit context list
- нет visual diff review
- нет pending changes buffer
- нет realtime token meter

Но это не первоочередной gap. Это слой после memory/spec/edit reliability.

## Что имеет смысл взять без ограничений по стоимости

### P1. Documentation Compiler + Spec Layer

Это самый правильный следующий шаг.

Нужно:
- довести `ai:docs:compile`
- сделать `feature dossier` canonical unit
- собирать его из Jira + repo + Hub artifacts
- использовать dossier как upstream для `jira-take`, aggregate docs и NotebookLM

Практический эффект:
- меньше ambiguous prompts
- меньше повторных объяснений
- выше воспроизводимость
- документация становится частью поставки

### P2. Semantic Memory Layer

Нужен не просто RAG по кускам текста, а memory model:
- `fact`
- `decision`
- `episode`
- `openQuestion`
- `evidence/sourceRef`

Желательно:
- hybrid exact + semantic search
- typed retrieval
- explicit save/load hooks
- source grounding

Если без экономии, можно смотреть в сторону shared vector DB / Graph-RAG поверх typed records.

### P3. Structured Edit Protocol

Добавить слой вроде hashline:
- file read возвращает line/block anchors
- модель редактирует anchors, а не "угадывает старый текст"
- patch validator отклоняет правку до записи, если anchors устарели

Это даст:
- меньше malformed edits
- меньше retries
- меньше output tokens
- лучшую совместимость между моделями

### P4. Critique Ranking

Следующий шаг после current agreement score:
- score critique по specificity
- повышать вес замечаний с file refs / code refs / concrete symbols
- понижать вес общих "looks fine" / "LGTM"-подобных реплик
- использовать ranking в consensus и `result-warning`

### P5. NotebookLM только как downstream surface

Идею из обсуждений стоит сохранить:
- canonical knowledge живёт локально в `.ai/docs/`
- NotebookLM получает только curated outputs
- raw logs / raw prompts туда не идут

## Что имеет смысл взять в cost-aware режиме

### P1. Local-first memory MVP

Самый практичный вариант по мотивам `994618` и `1001368`:
- SQLite + Markdown как canonical storage
- FTS5 для exact search
- `sqlite-vec` или простой local vector layer для semantic lookup
- локальные эмбеддинги только если реально нужны

Это дешевле и ближе к нашей архитектуре, чем сразу cloud Graph-RAG.

### P2. Deterministic docs/spec pipeline before LLM-heavy memory

Сначала стоит строить:
- dossier compiler
- source normalization
- evidence/sourceRefs
- project-level compiled docs

И только потом усиливать это model-heavy summarization.

Это даст более дешевый и более auditable baseline.

### P3. Heuristic review scoring вместо ML-модели

По мотивам `1000718` не нужно сразу обучать классификатор.

Достаточно rule-based scoring:
- есть ли code refs
- есть ли file/symbol refs
- есть ли question/suggestion
- есть ли конкретная failure mode
- это empty approval или нет

Это дешёво и уже сильно улучшит critique quality.

### P4. Structured edit protocol как selective mode

Не обязательно включать hashline везде сразу.

Рациональный вариант:
- large diffs
- retry after failed replace
- providers/models с плохой edit reliability

Так мы получим пользу без тотального усложнения пайплайна.

### P5. Complexity-based routing

Это логичное cost-aware развитие поверх уже существующей архитектуры:
- маленькие safe tasks -> reduced panel / single strong model
- risky or architectural tasks -> full debate

Это не главная идея статей, но это лучший способ монетизировать уже реализованные `prompt-gate`, cache и quality pipeline.

## Addendum: `1005610` — grep-ast instead of raw tree-sitter / vector search

Новая статья полезна именно для нашего вопроса про `tree-sitter`.

Главный вывод из неё:
- `tree-sitter` хорош как низкоуровневый AST-parser, но сам по себе плохо решает
  задачу **точного структурного поиска в реальной прикладной кодовой базе**;
- особенно плохо это проявляется там, где важна не "чистая грамматика языка", а
  семантика фреймворка, обёртки, DSL, annotations, metaprogramming и project
  conventions;
- в таких случаях либо приходится городить большой слой project-specific rules,
  либо точность начинает плавать от проекта к проекту.

Почему это очень похоже на нашу ситуацию:
- мы уже используем `tree-sitter` как optional AST/index layer;
- но не довели его до product-quality targeted extraction;
- именно потому, что "синтаксическое дерево" ещё не означает "умно найденный
  нужный кусок кода для ответа агента".

### Что статья меняет для нашего roadmap

Она подсказывает, что нам, возможно, не нужно дальше инвестировать в
`tree-sitter` как главный engine для answer-depth.

Более сильный вариант:
- оставить текущий code index как базовую инфраструктуру;
- для **structural search / targeted extraction / code discovery** попробовать
  `grep-ast`-подобный слой вместо попытки самим дотянуть tree-sitter до
  framework-aware поиска.

### Мой вывод по замене

Не стоит делать "полную замену tree-sitter на что-то другое" одним махом.

Правильнее:
1. перестать считать `tree-sitter` будущим answer-depth engine;
2. оставить его только как optional low-level parser/index helper;
3. отдельно проверить `grep-ast`-style structural search как новый engine для:
   - semantic file selection follow-up;
   - targeted method/class extraction;
   - critique-driven context expansion.

### Практический verdict

- **Да, новая статья даёт сильный аргумент против того, чтобы продолжать
  развивать answer-depth именно вокруг tree-sitter.**
- **Нет, tree-sitter не надо прямо сейчас "вырывать с корнем".**
- Лучший путь: перевести его в статус инфраструктурного помощника и
  исследовать `grep-ast` как candidate replacement именно для слоя
  структурного поиска и extraction.

### Code review signal (`1000718`)

У нас уже есть critiques и agreement.
Но пока нет distinction:
- useful critique
- shallow critique
- empty approval

Здесь возможен недорогой win.

## Что пока не стоит ставить в приоритет

- Полноценный custom IDE/UI как в `1000976`.
- Сразу дорогой cloud Graph-RAG.
- Self-improving skills на патчах до того, как появится canonical dossier/spec layer.
- Сложная модель обучения review-quality вместо rule-based scoring.

## Recommended Rollout

### Phase 1

- `ai:docs:compile`
- canonical `feature dossier`
- heuristic critique scoring
- local memory MVP without cloud dependency

### Phase 2

- semantic memory with typed entries
- structured edit protocol / hashline mode

### Phase 3

- `jira-take` consumes dossier/spec
- project compiled docs
- optional NotebookLM sync

### Phase 4

- TUI/UI additions: diff review, explicit context list, token meter, pending changes

## Final Recommendation

Если выбрать только три вещи с максимальным ROI, я бы брал:

1. `ai:docs:compile` + canonical feature dossier.
2. Local-first semantic memory MVP.
3. Structured edit protocol for reliable file changes.

Если выбирать самый дешёвый и самый полезный старт:

1. docs compiler,
2. heuristic critique scoring,
3. SQLite/Markdown memory,
4. selective hashline mode.

Итоговая оценка:

- **Сейчас мы уже сильные в orchestration, governance и runtime reliability.**
- **Сейчас мы слабы в memory, spec-first artifacts и edit reliability.**
- **Именно туда и надо вкладывать следующий прирост эффективности.**

### Short Verdict

Свежий батч подтверждает: **главный следующий прирост для хаба лежит не в “ещё больше debate”, а в качестве evidence pipeline, memory discipline и feedback loops**.

По текущим задачам проекта самый сильный practical overlap такой:
1. `discovery / navigation` до основного reasoning;
2. `post-expansion evidence assembly` и явные handoff-артефакты;
3. `typed memory + better recall discipline`;
4. `granular eval/feedback taxonomy`;
5. `internal capability metadata` для routing и stack-aware behavior.

### Что из статей нам реально полезно

#### 1. `1012106`: docs-first + navigation + clean context discipline

Из статьи полезно:
- `Claude.md` как короткий entrypoint с ссылками, а не как dump всего знания;
- `Sequential thinking` для многомодульных задач;
- `Serena` как navigation layer;
- правило `one task = one context` и `clear` после задачи;
- role split по областям (`backend`, `frontend`, `database`).

Что взять нам:
- не `claude.md` как основной формат, а короткий human-readable слой поверх `stack-profile.json`;
- отдельную bounded `navigation/discovery` фазу перед proposals;
- более жесткий handoff между большими фазами вместо накопления сырой истории.

#### 2. `1011868`: capability/policy/cost contract

Из статьи про `A2A` / `AGP` полезно:
- `AgentCard` как machine-readable описание возможностей;
- `Task` и `Artifact` как явные базовые сущности;
- capability metadata;
- policy metadata;
- cost-aware routing.

Что взять нам:
- не строить сейчас внешний `A2A` слой,
- а завести **внутренний capability contract** для наших агентов/ролей/фаз:
  - capability
  - trust expectations
  - policy/safety constraints
  - cost band

Это хорошо ложится на:
- `stack-profile.json`
- `agents.json`
- `Stack-Aware Dynamic Skills`
- будущий provider-aware scheduling

#### 3. `ai-manual`: anti-patterns and practical system patterns

Из статьи реально полезно:
- антипаттерн `blind RAG`;
- `Hybrid Search + reranking`;
- `Query Expansion / Step-Back Prompting`;
- `Observability Chain`;
- `Router -> Specialist -> Synthesizer`;
- `Fallback Cascade`;
- запрет на `2000-line prompt monolith`;
- prompt versioning in Git;
- `dynamic context optimization`;
- `simulate doubt`.

Что взять нам:
- сильнее бить по `blind retrieval` после seam expansion;
- считать prompts/config/contracts versioned artifacts;
- развивать не просто telemetry, а quality observability;
- продолжать переносить логику из prompt в код и runtime policy.

#### 4. `1008652`: strongest practical article in the batch

Наиболее ценные для нас паттерны:
- `Evals`
- `Semantic Monitoring`
- `Synthetic Data Pipelines`
- `Data Flywheel`
- `Side-by-Side Evaluation`
- `Granular Feedback Collection`
- `Memory & Summarization`
- `Dynamic Few-Shot`

Что это значит для нас:
- пилоты надо превращать в классифицированный improvement loop;
- outcome feedback должен быть не просто thumbs up/down, а с категориями;
- live regressions надо видеть не только по latency/token metrics, но и по semantic drift of failures.

#### 5. `1010522`: autonomous agent architecture as cautionary reference

Из статьи полезно:
- event-driven architecture;
- `PriorityQueue`;
- независимые loops (`event`, `proactivity`, `thoughts`);
- hybrid memory (`SQL + vector + graph`);
- sandboxed execution через Docker;
- явное признание проблем `amnesia`, `ReAct loops`, unsafe execution.

Что брать нам:
- не весь автономный always-on agent model,
- а отдельные bounded идеи:
  - better memory tiering
  - explicit sandbox contracts
  - stronger separation between sensors/events and reasoning core

Что не брать:
- не превращать hub в постоянно живущий autonomous agent runtime; это другая продуктовая форма и другой risk surface.

### Где проект уже лучше

- Multi-agent debate, critique, consensus, DA и trust gating у нас уже инженерно зрелее, чем в большинстве свежих статей.
- `Evidence-Grounded Patch Mode` сильнее generic “critical stance”, потому что grounded/assumed уже разделены.
- `Context Pack` + `Semantic Context Bridge` уже глубже, чем просто docs-first, markdown-first или vanilla RAG.
- `checkpoint/resume`, runtime telemetry и phase gating у нас сильнее типового “вайбкодинг” workflow из `1012106`.

### Где мы слабее

#### 1. Navigation before reasoning

Статьи `1012106` и `1010522` подсветили то, что мы сейчас видим и в `nornick`:
- reasoning часто стартует до того, как собрана идеальная evidence base;
- отсюда `missing-file-anchor`, `unconfirmed-seam`, file-reader loops и disagreement.

#### 2. Memory discipline

У нас есть memory MVP, но пока он слабее layered memory thinking из `1010522` и weaker than what our own pipeline now needs for hard cases.

#### 3. Feedback taxonomy

`1008652` правильно давит на granular feedback. Нам этого не хватает для systematic quality learning.

#### 4. Capability metadata

У нас пока нет явного internal `AgentCard`-подобного слоя.
Это делает routing и stack-aware behavior менее прозрачными, чем могли бы быть.

### Рекомендации: max-quality

#### P1. Navigation / Discovery micro-phase

Самый сильный кандидат под текущие bottlenecks.

Перед proposals запускать bounded phase, которая:
- находит likely owners;
- собирает candidate seams;
- выделяет anchor-worthy ranges;
- формирует evidence-ready discovery artifact.

Это прямо бьет по:
- `missing-file-anchor`
- `unconfirmed-seam`
- uneven approval evidence

#### P2. Post-expansion evidence digest

После seam expansion нужен не просто новый большой круг,
а curated evidence digest:
- what new seams were fetched
- what was confirmed
- what remains uncertain
- which anchors support which claim

#### P3. Granular pilot feedback taxonomy

Категории вроде:
- `missing-file-anchor`
- `unconfirmed-seam`
- `substantive-assumptions`
- `routing waste`
- `review disagreement`
- `insufficient context precision`

Дальше эти категории должны уходить в roadmap/logs/metrics, а не только в narrative notes.

#### P4. Role-specific instruction bundles

Не generic prompt-only roles, а generated overlays:
- by stack
- by framework
- by language
- by test/runtime platform

Это логичное развитие для `Stack-Aware Dynamic Skills`.

#### P5. Internal capability contract

Ввести внутренний `AgentCard`-like metadata layer:
- capability
- policy
- cost band
- trust role
- preferred task shapes

### Рекомендации: cost-aware

#### P1. Discovery first, not more late rounds

Cheap discovery phase потенциально выгоднее, чем дорогой extra revision circle.

#### P2. Dynamic context optimization with anchors preserved

Не глобальный compaction, а selective handoff compression:
- только между major phases;
- только с сохранением evidence anchors;
- только там, где input pressure мешает качеству.

#### P3. Fallback cascade for stable-provider paths

Идея из `ai-manual` полезна: часть routing можно осознанно делать по cost/latency/reliability band, а не только по статическому model assignment.

#### P4. Prompt/version contracts as code

Промпты, contracts и quality heuristics должны жить как versioned code artifacts с eval loop, а не только как “промпт-текст”.
