## Что из статей нам реально полезно

### 1. Hashline Edit Protocol (995986)

**Суть:** Каждой строке файла присваивается короткий хеш-тег (2-3 символа). Модель ссылается на хеш вместо угадывания точного текста. Результат: точность правок Grok Code Fast с 6.7% до 68.3%, снижение output-токенов на 61%.

**Применимость к нам:** Высокая. У нас сейчас правки основаны на str_replace / diff, что чувствительно к формату ответа модели. Hashline даёт:
- pre-apply reject по несовпадению anchors (файл изменился — правка отклоняется до применения)
- меньше retries при неточных правках
- provider-agnostic: работает на всех моделях одинаково
- экономия токенов

### 2. Spec-Driven Development (995466)

**Суть:** Цикл `intent → spec → plan → tasks → implementation → validation`. Спецификация — версионируемый артефакт намерения, хранится в repo, ревьюится, превращается в критерии приёмки.

**Применимость к нам:** Средне-высокая. У нас есть элементы (prompt-gate, ai:init, планы по docs:compile), но нет обязательного spec layer. Внедрение canonical feature dossier как upstream для jira-take — правильный следующий шаг.

### 3. Три типа памяти для агентов (994618)

**Суть:** Разделение памяти на три слоя:
- Fast facts (Redis) — имя проекта, текущая задача, ключевые решения
- Semantic search (ChromaDB) — поиск по смыслу прошлых разговоров
- Documents (Markdown) — архитектурные решения, чеклисты

**Применимость к нам:** Высокая. У нас сейчас есть typed logs и ai:memory snapshots, но это не memory system. Трёхуровневая архитектура — хороший blueprint для нашего memory layer.

### 4. Graph-RAG семантическая память (996154)

**Суть:** SPO-модель (Subject-Predicate-Object), pgvector + FTS, гибридный поиск, автоматическое выделение сущностей и фактов из текста.

**Применимость к нам:** Средняя (на перспективу). Сейчас для нас overkill — нам нужен сначала local-first MVP. Но typed facts с sourceRefs и hybrid retrieval — правильная целевая архитектура.

### 5. Scoring полезности code review (1000718)

**Суть:** 6 простых признаков (длина, вопросы, ссылки на код, альтернативы, пустые одобрения, специфичность) → логистическая регрессия → 87% точности в отделении signal от noise. Побочный эффект: модель предсказывает выгорание/уход разработчиков.

**Применимость к нам:** Высокая и дешёвая. У нас уже есть critique phase и agreement score. Добавить rule-based scoring по 6 признакам можно без ML — это даст фильтрацию shallow/empty critiques и повысит качество consensus.

### 6. EchoVault — локальная разделяемая память (1001368)

**Суть:** MCP-сервер, SQLite + Markdown, FTS5 + sqlite-vec с локальными эмбеддингами через Ollama. Память общая между агентами, ничего не уходит в облако.

**Применимость к нам:** Высокая как reference implementation для нашего memory MVP. Особенно полезна идея директивного тона в tool descriptions для повышения частоты сохранения контекста.

### 7. AI Factory (995038)

**Суть:** Автоматическая инициализация проекта, system of patches для обучения на ошибках, структурированные команды (task/feature/fix/evolve), DESCRIPTION.md как live spec.

**Применимость к нам:** Средняя. У нас ai:init уже сильнее. Но две идеи полезны:
- Patch-based learning loop (каждый фикс → документ → следующие задачи учитывают)
- Live project DESCRIPTION как обязательный артефакт

### 8. ProTalk Codex — IDE с AI (1000976)

**Суть:** Самодельная IDE с Monaco Editor, SSH-прокси, AI-агент с plan/patch/undo, explicit context panel, pending changes buffer.

**Применимость к нас:** Низкая на данном этапе. Но три UX-идеи стоит запомнить:
- explicit context list (что видит модель)
- pending changes buffer (буфер между агентом и файловой системой)
- visual diff review перед применением

### 9. Промпт-инженерия для кода (1000980)

**Суть:** Формула структурированного промпта: Роль + Контекст + Цель + Ограничения + Формат вывода. "Усилители" для ограничений по категориям.

**Применимость к нас:** Низкая. Это базовый уровень prompt engineering, который у нас давно покрыт prompt-gate и PROTOCOL.md. Полезно как напоминание о negative prompts (что НЕ делать).

### 10. GPT с нуля для Python-кода (995838)

**Суть:** Учебный проект создания LLM: кастомный токенизатор с AST, 10M параметров, контролируемая генерация. Ценность — в понимании внутренностей.

**Применимость к нас:** Нулевая для продукта. Это чисто образовательная статья, не влияющая на наш пайплайн.

## Где проект уже сильнее

1. **Multi-agent debate/critique/consensus.** Ни одна статья не описывает production-ready multi-stage pipeline с disagreement handling. У нас это ядро.

2. **Runtime hardening.** prompt-gate, response-validator, END OF DOCUMENT marker, checkpoint resume, result cache, delta context, adaptive context budget, quality feedback loop — всё это нас отличает от ad-hoc решений из статей.

3. **Project-scoped hub + strict dispatcher.** Дисциплина входа, project resolution, audit trail — у нас это жёстче, чем у любого решения из списка.

4. **Typed logs и reproducibility.** Явные артефакты, structured log entries с metadata — мы лучше решений, завязанных на состояние чата или UI.

5. **Context pack / code index.** Hybrid indexing с tree-sitter fallback уже сильнее, чем ручное управление контекстом в ProTalk Codex или полное отсутствие индексации в AI Factory.

6. **Test gate.** Обязательный `npm run ai:test` после изменений — этого нет ни в одной статье.

## Где мы слабее

1. **Память.** Наш самый серьёзный gap. Нет semantic retrieval, нет typed memory model, нет cross-session durable memory, нет общего memory API. Три статьи (994618, 996154, 1001368) здесь объективно впереди.

2. **Spec-first workflow.** Есть элементы, но нет обязательного потока `intent → spec → plan → tasks → code → validation`. Статья 995466 и частично 995038 показывают более зрелый подход.

3. **Structured edit protocol.** Правки чувствительны к формату ответа модели. Нет anchor-based editing, нет pre-apply reject, нет fallback matrix по моделям. Статья 995986 даёт конкретное решение.

4. **Critique quality weighting.** Есть agreement score, но нет ранжирования полезности замечаний. Статья 1000718 показывает, что rule-based scoring по 6 признакам даёт 87% точности.

5. **Patch-based learning.** Нет механизма сохранения уроков из фиксов для предотвращения повторения ошибок. AI Factory (995038) предлагает рабочий паттерн.

## Рекомендации: max-quality режим (без ограничений по стоимости)

### P1. Semantic Memory Layer с typed entries
- Typed memory model: `fact`, `decision`, `episode`, `openQuestion`, `evidence/sourceRef`
- Hybrid exact + semantic search
- Graph-RAG поверх typed records (по мотивам 996154)
- Shared vector DB с pgvector или Qdrant
- Explicit save/load hooks в пайплайне
- Source grounding для каждого факта

### P2. Documentation Compiler + Canonical Spec Layer
- Довести `ai:docs:compile`
- `feature dossier` как canonical unit
- Сборка из Jira + repo + Hub artifacts
- Dossier как upstream для jira-take и aggregate docs
- Обязательный spec review перед началом implementation

### P3. Structured Edit Protocol (hashline-level)
- File read возвращает line/block anchors с хешами
- Модель редактирует по anchors, не угадывает текст
- Patch validator отклоняет правку до записи, если anchors устарели
- Provider-specific fallback matrix (разные модели → разные edit formats)
- Экономия output-токенов до 61%

### P4. Critique Quality Ranking
- Score каждого critique по: specificity, code refs, file refs, concrete suggestions, severity
- Weighted consensus: качественные замечания влияют сильнее
- Автоматическая фильтрация "LGTM" / empty approvals
- Использование ranking в result-warning

### P5. Patch-Based Learning Loop
- Каждый фикс → structured document (проблема, причина, решение, профилактика, теги)
- Следующие задачи читают patch archive перед выполнением
- Накопление institutional knowledge в structured form

## Рекомендации: cost-aware режим

### P1. Local-first Memory MVP (самый приоритетный)
- SQLite + Markdown как canonical storage
- FTS5 для exact search
- sqlite-vec или простой local vector layer для semantic lookup
- Локальные эмбеддинги через Ollama только если реально нужны
- Без cloud-зависимостей
- Reference: EchoVault (1001368)

### P2. Heuristic Critique Scoring (самый дешёвый win)
- Rule-based scoring без ML:
  - Есть ли code refs (backticks, file extensions)?
  - Есть ли file/symbol refs?
  - Есть ли вопрос или предложение альтернативы?
  - Есть ли конкретная failure mode?
  - Это empty approval (LGTM/OK/+1) или нет?
  - Длина комментария
- Порог: <3 баллов → low-weight critique
- Интеграция в consensus и result-warning

### P3. Deterministic Docs Pipeline (before LLM-heavy memory)
- Dossier compiler
- Source normalization
- Evidence/sourceRefs
- Project-level compiled docs
- Дешевле и более auditable, чем model-heavy summarization

### P4. Selective Hashline Mode
- Не включать везде сразу
- Только для: large diffs, retry after failed replace, models с плохой edit reliability
- Постепенное расширение по мере проверки на production

### P5. Complexity-Based Routing
- Маленькие safe tasks → reduced panel / single strong model
- Risky or architectural tasks → full debate
- Логичное cost-aware развитие поверх prompt-gate, cache и quality pipeline

## Final Recommendation

**Три вещи с максимальным ROI без ограничений по бюджету:**
1. Typed semantic memory layer (fact/decision/episode + hybrid search)
2. Structured edit protocol (hashline anchors + pre-apply validation)
3. Canonical spec/dossier layer

**Три вещи с максимальным ROI в cost-aware режиме:**
1. Heuristic critique scoring (6 правил, без ML, немедленный эффект)
2. SQLite/Markdown memory MVP (EchoVault-подобный)
3. Selective hashline mode (только для проблемных случаев)

**Итого:**
- Наша orchestration, governance и runtime reliability — на уровне или выше всех описанных решений.
- Наши gaps: memory, spec-first artifacts, edit reliability, critique quality weighting.
- Именно туда нужно направлять следующий прирост эффективности.
- Совпадение с выводами Codex по направлениям ~90%, расхождения в деталях приоритизации и тактике.
