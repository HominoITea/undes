# User Agreement and Output Disclaimer

Status: draft internal baseline  
Scope: OSS/core usage notice until a release-ready legal package is finalized

This document defines the current responsibility boundary for using outputs
generated by the hub.

It is a product-facing usage notice, not jurisdiction-specific legal advice.
Before any public or commercial release, this text should be reviewed by legal
counsel for the target jurisdiction and business model.

## 1. Nature of the product

The hub is an AI-assisted engineering orchestration tool.

It may generate:

- analysis,
- code suggestions,
- configuration suggestions,
- architectural guidance,
- review comments,
- tests,
- and implementation drafts.

These outputs are provided as engineering assistance, not as a guaranteed
correct, complete, secure, or compliant final implementation.

## 2. Result contract for operators

The operator must read the trust header at the top of `.ai/prompts/result.txt`.

Current meanings:

- `RESULT_MODE: DIAGNOSTIC`
  The output is not verified as copy-paste-safe. It may still be useful, but it
  must be treated as advisory material only.

- `RESULT_MODE: PATCH_SAFE`
  The run passed the stricter patch-safe gate for that artifact. Even then,
  project-specific review and validation remain recommended.

- `COPYPASTE_READY: NO`
  Manual review, project-specific validation, and testing are mandatory before
  using generated code or configuration.

- `COPYPASTE_READY: YES`
  The artifact passed the current patch-safe gate, but the operator still owns
  final review, testing, and release responsibility.

If `.ai/prompts/result-warning.txt` exists, it must be treated as a required
companion file to the result.

## 3. Mandatory human validation

Before generated code, config, infra, or commands are used in any real project,
the user is responsible for:

- reviewing the output against the target codebase,
- checking symbols, types, methods, files, and seams,
- validating security, secrets handling, and permissions,
- running tests and additional project-specific checks,
- confirming compatibility with deployment and rollback expectations,
- obtaining any internal approvals required by the organization.

No generated output should be treated as safe for production use without such
validation.

## 5. Responsibility boundary

The user remains solely responsible for:

- deciding whether to use generated outputs,
- verifying them before application,
- testing them in the target environment,
- and accepting any consequences of applying them without adequate review.

If the user applies generated code, configuration, infrastructure changes, or
commands without review and validation, they do so at their own risk.

To the maximum extent permitted by applicable law, the maintainers are not
liable for losses, damages, incidents, outages, regressions, security issues,
or downstream consequences resulting from the use or misuse of generated
outputs, especially where those outputs were applied without independent review,

If a separate written commercial agreement exists, that agreement governs to
the extent it overrides this baseline notice.
